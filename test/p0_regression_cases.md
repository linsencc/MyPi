# MyPi P0 回归测试案例

本文档用于记录和指导 AI 在发版前进行 P0 级别的回归测试（Regression Test）。所有的测试验证均可以通过控制台日志、前端页面行为以及后端 API 的返回值进行断言。

## 1. 预期区域正常展示（预览区域刷新的正确时机）
**目标**：验证场景触发后，预览区域（或物理画框）能在预期的时间点准确完成刷新，画面内容覆盖正确。
* **TC-1.1 准时触发与覆盖**：创建一个即将在 1 分钟后触发的场景。观察控制台日志与预览区域。
  * **预期**：到达指定时间后，后台编排器（Orchestrator）立即捕捉到事件，并推送到渲染管线（Pipeline），最终预览区域展示对应的模板画面，刷新时机准确。
* **TC-1.2 画面完整性**：验证上屏的画面没有出现分辨率异常、排版错乱、字体丢失等渲染问题。

## 2. 时间线状态正常展示（过去与未来的调度）
**目标**：验证 Web 控制台对于“已展示”与“待展示”任务的时间轴解析和展示是准确无误的。
* **TC-2.1 即将展示（Upcoming）**：配置一个间隔周期（Interval）场景和一个固定时间（Cron）场景。
  * **预期**：时间轴要展示即将要展示的信息。能够清晰看到这几个场景下一次即将渲染的时间点，且计算结果符合当前时区（如东八区）。
* **TC-2.2 过去展示记录**：查看系统运行状态或日志记录（如 `WallRun` 记录）。
  * **预期**：前端能查看到过去刚刚执行过的场景记录，且展示的执行耗时、完成时间准确。

## 3. 绘画模版立即渲染与场景创建
**目标**：验证开发者提供的底层模板可以被正常实例化，并在前端即时反馈渲染结果。
* **TC-3.1 立即渲染（Live Preview）**：在 Web 端的“绘画模版”面板，选中任意一个模板（如 `daily_motto`），点击“立即渲染”。
  * **预期**：后端正常调用该模板的 `render` 方法，并在前端弹窗中快速展示渲染出的图片（Mock 数据测试），无报错。
* **TC-3.2 根据模板创建场景**：点击“创建场景”，输入特定的场景名称与模板自定义参数（如一段特定的寄语），保存。
  * **预期**：场景能够成功存入存储层（如 `storage/stores.py`），并在“场景管理”列表中出现，且携带有刚才填写的参数信息。

## 4. 场景渲染调度逻辑（周期循环、定时器及并发冲突）
**目标**：测试调度的核心引擎处理各种复杂时间规则及排队的健壮性。
* **TC-4.1 频率循环（Interval）正常工作**：配置一个每 5 分钟执行一次的场景。
  * **预期**：等待 5 分钟，确认其能够自动被调度并渲染上屏。
* **TC-4.2 周期定时（Cron）正常工作**：配置一个每周一至周五，早上 09:00 触发的场景。
  * **预期**：系统计算出的 `next_fire_time` 是正确的下个工作日 09:00。
* **TC-4.3 多场景并发设置与冲突解决**：同时设置两个场景在同一时间（例如都是今天 12:00）触发。
  * **预期**：系统不会发生死锁或画面穿插错乱。编排器的调度规则应当正常将它们排队处理，或根据规则只显示优先级高的那个。

## 5. 场景管理（状态开关、编辑、删除）
**目标**：测试场景的基本 CRUD 动作，验证后端状态持久化与前端页面刷新的联动。
* **TC-5.1 状态启停（Enable/Disable）**：在列表中点击某个正在排队场景的“启用/停用”开关（Switch）。
  * **预期**：停用后，该场景立即从“即将要展示的”时间队列中移除（调度取消）；再次启用后，重新计算并加入队列。
* **TC-5.2 编辑场景（Edit）**：修改某个场景的触发规则（比如从间隔 5 分钟改为 10 分钟）或修改它的文本参数。
  * **预期**：保存后，后端调度器即时重载该任务的时间，且下次触发时应用新的参数画面。
* **TC-5.3 删除场景（Delete）**：将一个场景彻底删除。
  * **预期**：从列表中消失，后台的定时任务队列中不再有它的残留。
