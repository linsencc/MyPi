# 电子水墨屏项目需求说明

## 1. 这份文档解决什么问题

本项目用于内网环境下自用，不考虑多用户、权限、租户等复杂场景。  
目标是把电子水墨屏内容展示流程做成“可配置、可扩展、可稳定运行”的画面单元系统。

一句话说明：  
在 UI 里配置画面单元和刷新时间，系统自动生成图片并刷新到水墨屏；后续新增图片类型时，只新增图片生成画面单元即可。

## 2. 业务边界（避免跑偏）

### 2.1 本期必须做

- 在 UI 中管理画面单元（新增、编辑、启停、删除、复制）。
- 每个画面单元独立设置刷新时间。
- 图片画面单元支持输入参数配置。
- 支持新增不同类型的图片生成画面单元，且不改调度核心。
- 记录画面单元运行结果，方便排查问题。

## 3. 核心需求（按使用者视角）

### 3.1 我可以单独设置每个画面单元的刷新时间

每个画面单元都要有自己的调度配置：

- `是否启用`：开/关
- `刷新方式`：手动 / 间隔刷新（后续可加 cron）
- `间隔秒数`：例如 300 秒
- `时区`：默认 `Asia/Shanghai`

### 3.2 我可以设置图片画面单元的输入参数

每个图片画面单元要声明自己的参数定义，UI 根据定义自动渲染表单。  
这样新增图片画面单元时，不需要单独开发一套新页面。

参数定义最少包含：

- 参数名（key）
- 类型（字符串、数字、布尔、枚举等）
- 是否必填
- 默认值
- 取值约束（最小值、最大值、格式）
- 说明文字

常见参数示例：

- 画布尺寸、方向
- 主题、字体
- 数据来源地址
- 图像处理参数（抖动、对比度、锐化）

### 3.3 我可以在 UI 中管理画面单元

UI 至少要提供以下能力：

- 画面单元列表：画面单元名称、画面单元类型、启用状态、下次刷新时间、最近执行状态。
- 画面单元操作：新增、编辑、启停、删除、复制。
- 运行记录：开始时间、结束时间、耗时、错误信息、输出文件路径。

### 3.4 我可以平滑扩展新的图片类型

系统采用“插件化图片画面单元”设计。  
新增一个图片类型时，只做两件事：

1. 新增一个图片生成画面单元实现执行逻辑。
2. 注册该画面单元的元信息（名称、参数定义）。

必须满足：

- 调度逻辑无需修改。
- UI 管理页无需新增专用页面。

## 4. 建议的数据模型（便于研发实现）

- `画面单元类型`：描述一个画面单元“能做什么”和“有哪些参数”。
- `画面单元实例`：描述一个具体画面单元“当前怎么配置、是否启用、多久执行一次”。
- `运行记录`：记录每次执行的状态、耗时、错误和输出结果。
- `画面单元连接`（可选）：用于表达上游输出给下游输入。

## 5. 水墨屏项目的硬约束

这些约束必须在实现中落地：

- 输出图尺寸必须匹配屏幕分辨率。
- 输出图颜色模式必须匹配屏幕能力（黑白/灰阶）。
- 支持全刷/局刷策略，减少残影问题。
- 下发前必须校验图片格式与尺寸，不合法要明确报错。
- 相同内容建议去重，避免无意义重复刷屏。

## 6. MVP（第一阶段）交付范围

第一阶段只做最关键闭环：

1. UI 画面单元管理（新增、编辑、启停、删除）。
2. 画面单元独立间隔刷新（先不做 cron）。
3. 1 个基础图片生成画面单元 + 1 个屏幕输出画面单元。
4. 执行记录与错误查看。

## 7. 验收标准（是否完成看这里）

满足以下即视为达标：

- 可以在 UI 创建多个画面单元，并为每个画面单元设置不同刷新间隔。
- 画面单元按各自时间稳定执行。
- 图片画面单元参数可在 UI 配置并生效。
- 新增一个图片生成画面单元时，不需要修改调度核心。
- 执行失败时可在 UI 中看到清晰错误信息。
